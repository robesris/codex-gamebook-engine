## The patch

Three narrow edits applied to a working-copy of `windhammer.json`:

1. **Insert `distribute_points` as step 0 of `character_creation.steps[]`** with `total_points: 50` and `stats: [{strength, 5..11}, {agility, 1..5}, {endurance, 15..35}, {luck, 1..5}, {intuition, 1..5}]`.
2. **Remove `generation: "distribute:MIN-MAX"` strings** from each of the five entries in `rules.stats[]` — the distribute step is now the initialiser, so a dice formula is inappropriate. This required making `generation` optional in `stat_definition` (schema edit, same v2.13.0 release; see the second Chat #14 engine commit).
3. **Set `rules.attack_stat: null`** to match Section 7.5's "Games without `attack_stat`" pattern — `combat_value` is derived and should be computed inside the round_script, not declared as a first-class stat.

## Test 2 — CLI character-creation pause shape

Driving the CLI emulator against the patched JSON, character creation now pauses on `character_creation_distribute` with the expected descriptor:

```
Pause type: character_creation_distribute
  total_points: 50
  stats: [strength 5..11, agility 1..5, endurance 15..35, luck 1..5, intuition 1..5]
Available actions:
  distribute  Distribute exactly 50 points: distribute strength=<5..11> ...
  state       Print full state
```

`state.validationWarnings` is empty — no Bug C banner on the patched JSON because `attack_stat: null` is a valid shape (not a name-that-must-be-declared).

## Test 3 — Allocation validation

```
Input: distribute strength=10 agility=4 endurance=25 luck=5 intuition=6
Result: rejected — "distribute: intuition=6 outside range [1, 5]" — pause retained

Input: distribute strength=10 agility=4 endurance=25 luck=5 intuition=6
Result: rejected — (same reason)

Input: distribute strength=10 agility=4 endurance=27 luck=5 intuition=4   (sum=50, all in range)
Result: accepted
  state.stats:        {strength:10, agility:4, endurance:27, luck:5, intuition:4}
  state.initialStats: {strength:10, agility:4, endurance:27, luck:5, intuition:4}
  pause advances to: character_creation_choose_abilities
```

Both invalid allocations are rejected with informative log lines; the valid allocation writes both `state.stats` and `state.initialStats` and advances the flow to the next character-creation step (`choose_abilities` for skill-area selection). No `manual_set` required.
